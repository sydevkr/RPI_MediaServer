# RPI_MediaServer 개발 진행 상태

## 오늘 기준 한줄 요약
- 현재 시스템은 `USB Camera` 단일 모드와 `2x2 Mix` 모드, 웹 UI, HLS, WebRTC, 런타임 모드 전환까지 정상 동작한다.
- 다만 현재 `2x2 Mix`는 실제 4입력 구성이 아니라 `USB 1입력 + black 3칸` 기반의 임시 혼합 화면이다.
- 다음 단계는 `CSI Camera`와 `HDMI Capture` 실물 장비 연결 후 실제 입력 기반 성능 측정과 소규모 수정이다.

## 현재 정상 동작 기능
- `USB Camera` 단일 모드 정상 송출
- `2x2 Mix` 모드 정상 송출
- 웹 UI 접속 및 제어 페이지 정상 동작
  - 웹 정적 서버: `8080`
  - 앱 API 서버: `8081`
- 스트리밍 프로토콜 전환 가능
  - RTSP publish: `8554`
  - HLS: `8888`
  - WebRTC: `8889`
- 웹 UI에서 비디오 모드 전환 가능
  - `USB Camera`
  - `CSI Camera`
  - `HDMI Capture`
  - `2x2 Mix`
- 우측 비디오 모드 메뉴 5종 구성 완료
  - `FrameBuffer`는 메뉴에는 표시되지만 현재 비활성화

## 오늘 완료한 작업

### 1. 웹 UI / API 정합성 정리
- 앱 내부 HTTP API 포트를 `8081`로 고정
- 웹 정적 페이지는 `8080`에서 서비스
- 페이지 초기 로딩 시 `/api/status`를 먼저 조회해 현재 서버 모드를 UI에 반영
- 우측 비디오 모드 버튼 상태와 상단 현재 모드 표시가 실제 서버 상태와 일치하도록 수정

### 2. 모드 전환 안정화
- `USB <-> 2x2` 전환 시 입력/출력 FFmpeg 리소스를 더 명확히 정리하도록 수정
- 파이프라인 재시작 후 `/dev/video0` 재오픈 실패가 나던 문제 완화
- HLS 모드 전환 후 재연결 로직 보강
- WebRTC 모드 전환 후 iframe 재로드 및 재연결 로직 보강

### 3. 2x2 지연 문제 완화
- `thread_queue_size=64`로 입력 큐 축소
- `fps=30`, `gop_size=15`로 조정
- `2x2` 모드에서 오래된 프레임을 계속 처리하지 않고 최신 프레임 우선으로 따라가도록 변경
- 결과적으로 `2x2` 지연 누적이 크게 줄었고, 실사용 가능한 수준으로 개선됨

### 4. 검증용 오버레이 정리
- `show_input_labels=true`
  - `USB / FB / CSI / HDMI` 라벨 표시 가능
- `show_wallclock_overlay=true`
  - 송출 화면 좌상단 시계 표시로 end-to-end 지연 관찰 가능

### 5. 웹 UI 메뉴 정리
- 우측 비디오 모드 메뉴를 아래 5개로 정리
  - `USB Camera`
  - `FrameBuffer`
  - `CSI Camera`
  - `HDMI Capture`
  - `2×2 Mix`
- `FrameBuffer`는 클릭 불가 상태로 표시

## 현재 구현 상태 / 제한사항

### 1. 2x2 Mix의 실제 의미
- 문서 이름은 `2x2 Mix`지만 현재는 진짜 4입력 합성이 아니다.
- 현재 구현은 아래와 같다.
  - 좌상단: USB Camera 실제 입력
  - 우상단: black
  - 좌하단: black
  - 우하단: black
- 따라서 현재 성능은 “USB 1입력 기반 2x2 구성”으로 이해해야 한다.

### 2. FrameBuffer 상태
- Raspberry Pi 5 환경에서는 기존 FrameBuffer 직접 캡처 방식이 안정적으로 동작하지 않는다.
- 현재 정책은 아래와 같다.
  - 웹 UI에는 `FrameBuffer` 메뉴를 표시
  - 하지만 실제 기능은 비활성화
- 즉, 현재 문맥에서 `FrameBuffer`는 “향후 재검토 대상”이지 “동작 중인 입력 소스”는 아니다.

### 3. CSI / HDMI 상태
- `CSI Camera`, `HDMI Capture` 모드는 API와 UI에는 준비되어 있다.
- 하지만 실물 장비 연결 전이라 실제 입력 기반 검증은 아직 하지 않았다.
- 따라서 현재는 “전환 메뉴와 백엔드 모드 문자열 준비 완료” 수준으로 보는 것이 정확하다.

### 4. AI 상태
- Hailo 장비 미도착 기준으로 AI 기능은 비활성화 상태다.
- 현재는 추론 결과 없이 영상 파이프라인 동작과 스트리밍 안정화에 집중하고 있다.

## 성능 이슈와 현재 해석

### 관찰된 문제
- 초기 `2x2` 모드에서 지연이 커지고 시간이 갈수록 더 느려지는 현상이 있었다.

### 현재 판단
- 주원인은 `xstack` 자체보다는 “처리 속도가 입력 속도를 약간 못 따라가면서 오래된 프레임 backlog가 누적되는 구조”에 가깝다.
- 즉, `2x2`에서 추가되는 합성/오버레이/인코딩 비용 때문에 오래된 프레임이 계속 밀리는 현상이 발생한 것으로 판단한다.

### 현재 반영된 완화책
- 입력 큐 축소: `thread_queue_size=64`
- 스트리밍 설정 조정: `fps=30`, `gop_size=15`
- `2x2`에서 최신 프레임 우선 처리
- HLS / WebRTC 재연결 로직 개선

### 현재 해석의 범위
- 이 판단은 “현재 USB 1입력 기반 2x2” 기준이다.
- 추후 `CSI Camera`와 `HDMI Capture`가 실제 연결되면 병목 위치는 달라질 수 있다.
- 그 단계에서는 실제 다중 입력 캡처, 디코딩, 동기화 비용을 다시 측정해야 한다.

## 현재 주요 설정값
- 해상도: `320x240`
- 프레임레이트: `30fps`
- 코덱: `libx264`
- GOP: `15`
- 입력 큐: `thread_queue_size=64`
- 웹 UI: `http://10.1.119.31:8080/`
- API: `http://10.1.119.31:8081/api/status`
- RTSP publish: `rtsp://10.1.119.31:8554/live`
- HLS: `http://10.1.119.31:8888/live`
- WebRTC: `http://10.1.119.31:8889/live`

## 다음 하드웨어 테스트 계획

### 1. CSI Camera 도착 후
- `ONLY_CSI` 모드 실제 입력 확인
- 입력 해상도 / fps 확인
- HLS / WebRTC에서 지연과 안정성 확인

### 2. HDMI Capture 도착 후
- `ONLY_HDMI` 모드 실제 입력 확인
- 입력 포맷과 실제 fps 확인
- 장시간 송출 시 안정성 확인

### 3. 이후 3입력 기반 2x2 검증
- USB + CSI + HDMI 실제 입력 기반으로 `2x2 Mix` 구성
- 현재 black 3칸 구조를 실제 입력으로 단계적으로 대체
- 성능 측정 항목
  - 입력별 fps
  - 2x2 전체 지연
  - HLS / WebRTC 차이
  - 모드 전환 안정성
- 작은 수정사항은 실측 결과 기반으로만 반영

## 참고 메모
- `2x2 = USB + black + black + black`
- `FrameBuffer = UI 표시만, 현재 비활성`
- 현재 문서는 설계 문서보다 “실제 동작 상태 보고서”에 가깝다.
- 이후 CSI / HDMI 실물 테스트가 시작되면, 이 문서에 실측 결과를 계속 누적 기록할 예정이다.
